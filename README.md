# タスク管理アプリ(Todo App)

🔗 *[デモを見る](https://react-to-do-nu-khaki.vercel.app/)*

## 概要

タスクの追加・編集・削除・完了管理・期限設定・フィルタリングを備えたタスク管理アプリです。  
バニラJSによるフロントエンド単体実装からスタートし、React によるコンポーネント設計への移行、さらに Express + MongoDB を用いたフルスタック構成へと段階的に発展させました。

---

## なぜ作ったか

「どこを直せばいいか分からなくなる」という経験から、設計を意識したアプリ開発を学ぶためにこのアプリを作り始めました。  
機能追加のたびにコードが肥大化する問題に直面し、データ管理と UI の分離・コンポーネント設計・バックエンドとの連携という順序で課題を解決しながら開発を進めました。

---

## 開発の変遷

### v1：バニラ JS 版

**学んだこと**

- データ管理と UI 操作の責務分離（TaskManager / Storage クラス）
- 一方向のデータフローによる状態と UI の整合性の保ち方
- localStorage を用いたデータ永続化
- render 関数の分割による保守性の向上

**設計図**

```
ユーザー操作 → TaskManager（データ更新）→ Storage（localStorage 保存）→ render（UI 再描画）
```

**限界と次への動機**

クラス設計で責務分離はできたものの、DOM 操作のコード量が多く、状態変化のたびに全体を再描画する非効率さが残りました。また、機能追加のたびに render 関数の分割を手動で管理する必要があり、コンポーネント単位で UI を管理できる仕組みの必要性を感じ、React への移行を決めました。  
（※ v1 の README にも「今後の改善：React での再実装」として記載しています）

---

### v2：React 版

**Reactで解決したこと**

- コンポーネント単位（`App` / `TaskItem`）での UI 管理により、各タスクの編集状態を独立して保持できるようになった
- `useState` / `useEffect` による宣言的な状態管理で、データと UI の同期が自動化された
- DOM 操作のコードが不要になり、ロジックに集中できるようになった

**設計図**

```
App（状態管理・フィルター・CRUD ロジック）
 └── TaskItem × n（各タスクの表示・編集状態管理）
```

**限界と次への動機**

localStorage によるデータ管理はブラウザ依存であり、複数端末での利用やデータの永続的な保持ができませんでした。実用的なアプリに近づけるためにサーバーサイドとデータベースの導入を決めました。

---

### v3：React + Express + MongoDB 版

**バックエンド化で変わったこと**

- データが MongoDB に永続化され、ブラウザをまたいでタスクを管理できるようになった
- REST API（GET / POST / PUT / DELETE）を設計・実装し、フロントとバックエンドの責務を明確に分離した
- 更新可能フィールドをホワイトリスト方式で制限し、意図しないデータ変更を防ぐ設計にした
- バリデーション・エラーハンドリング・404 ハンドラを実装し、異常系にも対応した

**開発中に直面した問題と解決**

- **環境変数の管理**：MongoDB の URL をコードに直書きすると GitHub 経由でパスワードが流出するため、`.env` ファイルに切り出して管理した。フロントエンドの URL も `.env` で管理することで、ポート番号の変更を一箇所で済むようにした
- **CORS エラー**：サーバー側の CORS 設定が `http://localhost:3000` のままで、React が `http://localhost:5173` で起動していたため通信できなかった。フロントとバックエンドのポート番号を統一し、`FRONTEND_URL` を `.env` で管理することで解決した
- **フォルダ構成の複雑化**：フロントとバックエンドが別フォルダになったことでエラーの発見に時間がかかった。どちらのサーバーも同時に起動しながら動作確認するという開発フローに慣れるまで苦労した
- **MongoDB の日付型の不一致**：`deadline` フィールドを `new Date()` で変換していたため型が合わず不具合が発生。文字列のまま保持する方針に統一して解決した
- **非同期エラーの2種類への対処**：`await fetch` の失敗には「ネットワーク自体の切断（例外が投げられる）」と「サーバーは応答したが 500 エラー（`res.ok` が false になる）」の2パターンがある。`try/catch` と `if (!res.ok)` を組み合わせることで両方に対応した

**設計図**

```
React（フロントエンド）
 └── fetch（REST API）
      └── Express サーバー
           └── Mongoose（スキーマ・バリデーション）
                └── MongoDB
```

## 機能一覧

- タスクの追加 / 編集 / 削除
- 完了 / 未完了の切り替え
- 期限設定と期限に応じた表示変更（過去：赤、当日：オレンジ）

- データ永続化（v1：localStorage、v3：MongoDB）
- エラーハンドリング・バリデーション

---

## 工夫した点

- **設計の一貫性**：v1 から一貫して「データ操作と UI の分離」を意識し、各バージョンでその考え方を発展させた
- **ホワイトリスト方式**：PUT エンドポイントで更新可能フィールドを明示的に制限し、不正な更新を防いだ
- **異常系への対応**：空入力バリデーション・DB 接続エラー・404 ハンドラを実装し、想定外の状態でもアプリが破綻しないようにした
- **UX の一貫性**：タスク追加後にフィルターを「すべて」に戻すことで、追加したタスクが必ず表示される状態を保った
- **エラー通知**：通信失敗時に `alert` でユーザーに知らせることで、操作が成功したか失敗したかを明示した
- **runValidators の明示的な有効化**：MongoDB は更新時デフォルトでバリデーションが走らないため、`runValidators: true` を指定してスキーマのバリデーションを更新時にも適用した

## セットアップ方法

### v1（バニラ JS 版）

```bash
# ファイルをブラウザで開くだけで動作します
open index.html
```

### v2（React 版）

```bash
npm install
npm run dev
```

### v3（React + Express + MongoDB 版）

**バックエンド**

```bash
cd backend
npm install
# .env ファイルを作成
echo "MONGODB_URL=your_mongodb_url" > .env
echo "FRONTEND_URL=http://localhost:5173" >> .env
node server.js
```
